## 1. Executive summary

- **The rotational path token carries a forward-versus-reverse polarity inside the
  rotational-dex register.** The strongest corpus-supported reading is that
  `FRONT`/`BACK` marks that polarity: curator descriptions consistently align
  "whirling" with `FRONT` and "reverse whirling" / "reverse-swirling" with `BACK`,
  and PassBack names the reverse wrap "Twisting". This is *functionally analogous*
  to how `IN`/`OUT` works in the naming system. **The precise physical property
  carrying the polarity — passage around the leg, circle phase, body-relative
  orientation, or a transformed local `IN`/`OUT` — remains unresolved.**
- **Six of eight written sub-token combinations occur** (§3); leg, sense, and wrap
  labels are at least partly separable, but full independence is not established,
  and two combinations are absent.
- **Reducing the wrap to bare leg × IN/OUT collapses distinct canonicals** into
  existing base atoms — verified for `blink` against the exact mirage and illusion
  rows (§6). But a *string* collision after reduction does **not** disprove a
  transformed-coordinate reading; it may mean the body orientation needed to
  interpret the event was discarded (§5, hypothesis kept viable).
- **Two separate sufficiency judgments (§10).** *Operationally*, the full legacy
  register (leg + front/back + wrap) is discriminatively sufficient for the stored
  rows. *Explanatorily*, the notation is **not yet understood** well enough to be
  called sufficient: A3 has not established what `WHIRL` vs `SWIRL` contributes once
  the leg is written separately, nor whether `FRONT`/`BACK` is primitive or
  derivable from carried movement state.
- **The v1.2 notation models the path as dex-local** (bracketed on `[DEX]`,
  recurring ahead of several terminal classes), so a portable-dex-property reading
  is the best current representation — but token-by-token portability is **uneven**
  (`FRONT WHIRL` is broad; `BACK SWIRL` occurs only in `blink`), so physical
  independence from the receiving structure stays a hypothesis (§9).
- **Twirl:** the v1.2 representation strongly supports a **path-bearing dex feeding
  an osis engine** — its `base_trick = osis`, its formula, its curator description
  ("a swirling osis"), and PassBack's technical name ("Twisting near Osis") all
  agree. Whether players perceive and name Twirl as an independently bundled
  movement is a physical/historical question stored structure alone cannot close.
- **Evidence addendum (§8)** surfaces the strongest new Observed evidence: the
  reverse-swirl family (blink / revup / twirl) is one wrap ("Twisting") named by
  its terminal (near Toe / near Pickup / near Osis), and the Polini reverse-swirl
  tutorials are referenced but **not ingested**, so their physical content is not
  yet extractable.

The full active corpus was searched for the literal path tokens `FRONT SWIRL`,
`BACK SWIRL`, `FRONT WHIRL`, `BACK WHIRL` in `operational_notation`. **48 active
rows / 49 occurrences** (`reverse_whirling_twirl` carries two). Frequency:

## 3. [Observed → Derived] Sub-token structure and the combination gap

Each occurrence is written `<leg: SAME|OP> <sense: FRONT|BACK> <wrap: WHIRL|SWIRL> [DEX]`.
With three binary sub-parts there are **eight** possible written combinations. The
corpus shows **six**:

| present (6) | absent (2) |
|---|---|
| OP FRONT WHIRL, SAME FRONT WHIRL, OP BACK WHIRL, SAME FRONT SWIRL, OP FRONT SWIRL, OP BACK SWIRL | **SAME BACK WHIRL, SAME BACK SWIRL** |

**[Derived].** Six of eight occur, so leg, sense, and wrap labels are **at least
partly separable** in the notation — but the corpus does **not** establish their
complete independence. The two missing combinations (`SAME BACK WHIRL`,
`SAME BACK SWIRL`) may be accidental gaps, invalid cells, unnamed movements, or
evidence that some coordinates constrain others (e.g. a reverse wrap may correlate
with the OP leg). Recorded as an open question (§12), not resolved.

The **wrap word** (WHIRL/SWIRL) is written *alongside* an explicit leg, so it is
not encoding the leg here (unlike the base four-cell matrix). What it carries once
the leg is separate — plane, handedness, or inherited base-atom character — is
**unresolved** and is one of the explanatory gaps in §10.

## 5. [Derived] Competing hypotheses for FRONT / BACK

Observation to explain: `FRONT` attaches to the forward operators ("whirling",
"swirling"/twirl), `BACK` to the reverse operators ("reverse whirling",
"reverse-swirling"/blink), and PassBack calls the reverse wrap "Twisting".

| hypothesis | status |
|---|---|
| **(A) A forward/reverse polarity within the rotational register** | **Best supported.** Matches the descriptions and the Swirling(front)/Twisting(back) PassBack naming. Establishes a *polarity*, not the physical carrier. |
| (B) Anterior vs posterior passage of the wrapping leg | Possible physical carrier of (A); not excluded, not proven without footage. |
| (C) Leading vs trailing phase of a complete leg circle | Possible carrier of (A); indistinguishable from (B) on notation alone. |
| (D) Inside vs outside passage relative to the body | Weakly supported; overlaps (B) once the leg is written separately. |
| (E) Support-leg topology | Not the primary meaning; support is written by `(no plant while)` and varies independently. |
| (F) Body-facing orientation | Carried by `SPIN [BOD]`, separate from the wrap token. |
| **(G) A rotationally transformed IN/OUT** | **Still viable.** `FRONT`/`BACK` may encode a movement describable with `IN`/`OUT` only after carrying body orientation, support state, and an event-local reference frame. A text collision after reducing the token does **not** disprove this — it may mean the frame needed to interpret the `IN`/`OUT` was discarded. The current flat notation cannot test the reduction safely; the later state-transition model is meant to. |
| (H) Historical register with no independent axis | Rejected: `blink` and the reverse-whirling rows show the token does real distinguishing work in active canonicals. |

**[Derived].** The polarity (A) is the firm result; the physical carrier (B/C/D)
and the transformed-coordinate reading (G) are **all open**, and G is exactly what
the next study must adjudicate. A3 does not prejudge it.

## 6. [Derived] Reduction and collision — verified against canonical rows

`blink` = `SET > OP BACK SWIRL [DEX] > OP TOE [DEL]`. Reducing the wrap to a bare
linear dex, checked directly against the database:

| blink reduction | exact resulting formula | existing canonical occupant | information lost |
|---|---|---|---|
| OP BACK SWIRL → OP IN | `SET > OP IN [DEX] > OP TOE [DEL]` | **mirage** (ADD 2, exact match) | the reverse-swirl wrap; blink collapses into mirage |
| OP BACK SWIRL → OP OUT | `SET > OP OUT [DEX] > OP TOE [DEL]` | **illusion** (ADD 2, exact match) | the reverse-swirl wrap; blink collapses into illusion |

Both reductions land on an existing **base atom** — the wrap is exactly what keeps
`blink` distinct from mirage and illusion. (My earlier "pickup/mirage-shaped"
phrasing was wrong: pickup terminates `SAME TOE`, not `OP TOE`; the true occupants
are mirage and illusion, verified.) For the deeper compounds other brackets
(`[PDX]`, `SPIN`, duck) would still differ after reduction, so the whole row would
not merge — but the wrapping event's reading would still be corrupted. **[Ruled]
anchor:** the Notation paper already forbids the bare rotational dex because
mechanical translation "provably collapses distinct canonicals." §6 supplies the
exact instance. Crucially, this proves a *string/discrimination* loss; per §5(G)
it does not by itself decide whether a state-carrying model could recover an
`IN`/`OUT` reading.

## 9. [Derived] Where the path lives — dex-local, but uneven

The path token is bracketed on `[DEX]` and rotational tokens recur ahead of
multiple terminal classes, so **the v1.2 notation models the path as dex-local**,
and a portable-dex-property reading is the best current representation. But
portability is **token-by-token uneven** — the examples are not "one and the same
wrapping dex":

| exact token | clipper terminal | toe terminal |
|---|---|---|
| FRONT WHIRL | 28 | 10 |
| BACK WHIRL | 4 | 1 |
| FRONT SWIRL | 5 | 0 |
| BACK SWIRL | 0 | 1 (`blink`) |

**[Derived].** `FRONT WHIRL` is broadly distributed across terminals; `FRONT SWIRL`
appears only with the osis/clipper twirl family; `BACK SWIRL` occurs **only** in
`blink`. So physical independence of the wrap from its receiving transition is a
**hypothesis supported for the common tokens and untested for the rare ones**, not
settled doctrine. The dex-local model is the best representation; it is not proven
that every rotational wrap is terminal-independent.

## 10. [Derived] Sufficiency — two separate judgments

**Operational discrimination.** The full legacy rotational register (leg +
front/back + wrap) **is** discriminatively sufficient for the currently stored
rows: retaining all sub-tokens keeps the formulas distinct as strings and avoids
the reductions that collapse them into ordinary toe or clipper cells (§6). As a
collision-preventing notation it works.

**Explanatory sufficiency.** The notation is **not yet understood** well enough to
call it explanatorily sufficient. A3 has **not** established what `WHIRL` vs
`SWIRL` contributes once the leg is written separately (§3), nor whether
`FRONT`/`BACK` is a primitive axis or derivable from a carried movement state
(§5(G)). The register currently preserves distinctions Atlas cannot yet fully
explain. A black-box token that prevents collisions is useful notation; Atlas's
goal is to understand the movement information it carries, and that goal is not met.

**Consequently: no new token is recommended, and no conclusion is drawn that no
missing property exists.** Both questions stay open pending the physical-evidence
review and the state-transition study.

## 11. [Derived] Minimum state model for the next study

To normalise relational coordinates across events, the model must carry — and must
distinguish relational labels from **persistent physical identities**:

| state variable | why |
|---|---|
| **persistent physical left/right leg identity** | so "same physical leg-circle" is trackable across events — without it the key question below cannot be simulated |
| current support / plant state (incl. airborne / no-plant) | anchors "same side as plant"; `(no plant while)` changes it |
| current bag-contact / setting foot | which foot last touched or set the bag |
| current free / dexing leg | the leg performing the next event |
| body orientation | flipped by `SPIN [BOD]`; re-evaluates SAME/OP |
| event-local SAME/OP reference frame | the frame in which the next token's SAME/OP is read |
| bag position relative to body and dexing leg | for inside/outside and front/back reconstruction |
| rotation / wrap phase (where observable) | for the front/back anatomy question |
| intended receiving foot and surface | the terminal target |

**The question the model must answer:** *does the same physical leg-circle receive a
different SAME/OP label after the preceding event changes support or body
orientation?* This is the A2 anomaly (the "swirl" motion written `SAME OUT` leading
but `OP OUT` following a wrap/whirl dex). Without **persistent physical-leg
identity** the question is not even expressible, which is why it heads the list.

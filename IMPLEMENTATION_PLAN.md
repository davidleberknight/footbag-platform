# IMPLEMENTATION_PLAN.md

This is the single plan of record for the footbag.org modernization: every open work item and every open question we still need to resolve, grouped by who owns it. It is the clean baseline we will load into the go-live kanban board. Long-term design and rationale live under `docs/`; this file tracks what is left to do.

How to read it. Sections are ordered by owner, from the people we are coordinating with (Steve, then Julie) to the maintainers doing the build (Dave, then James, whose pipeline work carries the most detail). Each item is written to stand on its own: you should not need to open another document to understand what the task is. A reference to `docs/MIGRATION_PLAN.md` (the detailed migration design) or a file path appears only where it points at the actual work or saves copying a large table here. Finished items are deleted from this file rather than marked done.

Owners. Steve is the current footbag.org webmaster (he holds the legacy-site, DNS, email, and data-export facts). Julie is the IFPA secretary (she decides IFPA governance, groups, and votes). Dave is the project maintainer and manager (platform code, and he drives every open question to a decision). James is the historical-data and freestyle maintainer (the data pipeline and all freestyle content).

### Platform deviations and go-live gates

Each of these is a place where the running code differs from the intended design, or a launch requirement that is not built yet.

- **Internal QC tools must never reach production.** The operator-only QC tools under `src/internal-qc/` (reachable at `/internal/*` in development and staging) ship in the source but must not be reachable in production. Currently the production image build strips them out and replaces their routes with a dead stub. They are removed for good when the QC subsystem is retired.
- **Dev and staging get admin accounts automatically.** To avoid hand-writing SQL, registration in development and staging promotes a configured list of emails to admin (`applyDevStagingBootstrapAdmin`). The intended design has exactly one way to create the first admin: a one-time token claim at `/admin/bootstrap-claim`. Every shortcut here is tagged `CUTOVER-REMOVE` and is deleted at go-live (procedure in `src/dev-bootstrap/README.md`); until then, production is protected by fail-fast config checks and by the production image stripping this code out (the `CUTOVER-REMOVE` strip runs in all three Dockerfiles: `docker/web`, `docker/worker`, and `docker/image`).
- **Build the admin club-cleanup actions (to retire the override CSV).** Before go-live, club classification fixes (force-keep or force-junk a club) are hand-edited rows in a pipeline CSV, because the database is rebuilt from CSVs on every reload (James owns that CSV procedure, below). The real design is an admin work-queue action that does this against the live database. Building those admin actions retires the temporary CSV approach.
- **Seed the remaining test personas as their features get built.** The test suite seeds a sample user for every account type whose feature exists. The still-missing ones are blocked on features that do not exist yet: event organizers, group owners and members, vote-eligibility lists, and a banned legacy member. Add each persona in the same change that builds its feature.
- **Wire up the accessibility tests.** The testing standard calls for automated accessibility checks (using axe, against the WCAG 2.1 AA standard) on every important page. A few `@a11y`-tagged end-to-end checks exist for keyboard reachability and form-label association on the onboarding flow, but there is no axe-core WCAG scan on any page and `run_all_tests.sh` has no accessibility-only mode. Add axe-core checks across the browser test suite, tag the high-traffic public ones for the quick smoke run, and add an option to `run_all_tests.sh` that runs only them.
- **Source the Stripe webhook secret from AWS, not by hand.** The design says production reads the Stripe webhook signing secret from AWS Parameter Store at startup, like the other secrets. Currently the operator pastes it into the server's env file by hand because no Parameter Store entry exists for it. Add the secret to Parameter Store in both staging and production Terraform, grant the server permission to read it, have the deploy pull it from there, and update the DevOps guide.
- **Actually send the notifications the services promise.** Three services document, in their header contracts, emails and mailing-list updates that the code does not yet send: the membership-tiering service (tier-change and Hall-of-Fame / Big-Add-Posse congratulation emails, plus an admin-alerts mailing-list update when someone becomes admin), the active-player service (a vouch-confirmation email), and the event service (organizer, participant, and sanctioning emails). The email outbox and worker that send these already exist and are tested; the gap is that the service code never enqueues them. Add each send after its database transaction commits. The event-service emails arrive with the event-organizer feature.
- **Build the curator source-of-truth cutover tests.** At go-live the curator content source of truth switches from the `/curated/` JSON sidecars to the persistent production DB (the seeder stops running against production; admins author through the admin UI). Add special tests that pin this switch: curator `media_items`, galleries, and S3 bytes survive a data-preserving `scripts/deploy-migrate.sh` deploy with no seeder run; admin-UI create, edit, and delete is the sole authoring path and is durable without a seeder replay; and the seeder fails fast rather than orphan-deleting admin- and member-created rows if pointed at a persistent production DB. (Migration gate OR9, §29.14; DD §1.13.)
- **Harden the e2e stack against writing the real `/curated/` tree.** The e2e stack (`scripts/e2e/start-stack.sh`) boots `NODE_ENV=development` with `ALLOW_CURATED_SIDECAR_WRITES=1`, so the `getCuratedRootDir()` test-mode throw in `curatorMediaService` does not apply there. No current e2e test exercises a curator-admin write (the gallery and avatar tests use the member-self path, which never touches `/curated/`), but a future one would write into the committed `/curated/` working tree. Point the e2e stack's curated root at a temp directory (an override the service already honors) so an e2e curator write can never reach the real tree.

### Security and correctness fixes (from the adversarial bug sweep)

Each item is a verified defect with its file location and the fix. Each was re-confirmed against the current code. Delete the item when the fix lands with a regression test.

- **The required CI gate omits the Terraform job.** In `.github/workflows/ci.yml` the `terraform` job is defined and runs, but it is absent from the `ci-complete` aggregate gate's `needs` array (around line 231) and from its results-check block, so a PR with broken Terraform `fmt`/`validate` can merge to `main` while `ci-complete` reports success. Under the Terraform-only operating model this is the canonical infrastructure surface. Fix: add `terraform` to the `ci-complete` `needs` array and its results-check block. Touching `.github/` needs maintainer approval.
- **The Content-Security-Policy may block Turnstile when CAPTCHA goes live (latent).** In `src/app.ts`, `script-src` and `frame-src` allow `https://challenges.cloudflare.com` but `connect-src` (around line 95) is `'self'` plus the optional S3 origin only. If the managed Turnstile widget performs `fetch`/XHR to `challenges.cloudflare.com`, those calls are blocked and the widget silently breaks on login, register, password-reset, and legacy-claim once CAPTCHA is enabled. CAPTCHA is project-deferred and stubbed today, so this is not yet reachable; confirm against a live widget in a browser, and if confirmed add the domain to `connect-src`.
- **Strengthen the media and gallery tests against the gallery user stories.** The gallery and media route tests miss several cases the media galleries expose: a mixed photo-and-video tag browse (`POST /media/browse` currently seeds photos only), sort order actually reordering the rendered grid (only the stored `sort_order` is asserted, never the rendered order), a member-owned mixed gallery's item count on `/media/member-galleries`, and re-evaluating gallery membership after a criteria-tag edit. The partials also need positive video-tile assertions in a mixed grid, mixed-set prev/next, null-caption alt-text fallback, and per-platform thumbnails. Note the real markup before writing these: gallery grid tiles use `.gallery-tile-video` plus the `.gallery-tile-play` SVG, not `class="video-facade"`, and the `data-platform`/`data-embed-url`/`data-video-src` attributes live on the item-detail view (`src/views/media/gallery-item.hbs` via `src/views/partials/video-facade.hbs`), not on the grid tile; a grid test must assert the former. (V_View_Gallery, M_Organize_Media_Galleries, M_Upload_Photo, M_Submit_Video)

Once the worklist is resolved, the existing resolution runs against Steve's final dump; (c) is substantially complete: the read-only member-ID cross-check harness exists (`crosscheck_member_profile_ids.py`), and against the current data every profile-URL ID resolves to a member account and every result-attribution link is backed by a member row, leaving only a rerun with the member IDs taken from Steve's frozen export; and (d) load the final dump taken after the site is frozen. - **Optional: tidy a few dangling contact-label residuals in club descriptions.** The contact-detail scrub is settled: a full re-derivation of every description in `legacy_data/seed/clubs.csv` finds no email addresses and no phone numbers (the only digit runs are Facebook group IDs inside URLs, year ranges, and dates; the `@` characters mean "at a location" or sit at an already-stripped value). The `--from-csv` path reads `legacy_data/event_results/canonical_input/` as its source-data input, supplied on the maintainer machine; it stays gitignored because committing it (so a fresh clone could build with no mirror) trips the CI loader-smoke real-data guard, so that fresh-clone step is deferred to a gate change that stages the smoke fixture into a throwaway pipeline root. The two builds are identical except `net_review_queue`: a no-mirror build omits the QC-issue rows that only the backbone produces, an accepted and documented delta rather than a defect.

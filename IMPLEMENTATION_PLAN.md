# IMPLEMENTATION_PLAN.md

This is the single plan of record for the footbag.org modernization: every open work item and every open question we still need to resolve, grouped by who owns it. It is the clean baseline we will load into the go-live kanban board. Long-term design and rationale live under `docs/`; this file tracks what is left to do.

How to read it. Sections are ordered by owner, from the people we are coordinating with (Steve, then Julie) to the maintainers doing the build (Dave, then James, whose pipeline work carries the most detail). Each item is written to stand on its own: you should not need to open another document to understand what the task is. A reference to `docs/MIGRATION_PLAN.md` (the detailed migration design) or a file path appears only where it points at the actual work or saves copying a large table here. Finished items are deleted from this file rather than marked done.

Owners. Steve is the current footbag.org webmaster (he holds the legacy-site, DNS, email, and data-export facts). Julie is the IFPA secretary (she decides IFPA governance, groups, and votes). Dave is the project maintainer and manager (platform code, and he drives every open question to a decision). James is the historical-data and freestyle maintainer (the data pipeline and all freestyle content).

### Platform deviations and go-live gates

Each of these is a place where the running code differs from the intended design, or a launch requirement that is not built yet.

- **Internal QC tools must never reach production.** The operator-only QC tools under `src/internal-qc/` (reachable at `/internal/*` in development and staging) ship in the source but must not be reachable in production. Currently the production image build strips them out and replaces their routes with a dead stub. They are removed for good when the QC subsystem is retired.
- **Dev and staging get admin accounts automatically.** To avoid hand-writing SQL, registration in development and staging promotes a configured list of emails to admin (`applyDevStagingBootstrapAdmin`). The intended design has exactly one way to create the first admin: a one-time token claim at `/admin/bootstrap-claim`. Every shortcut here is tagged `CUTOVER-REMOVE` and is deleted at go-live (procedure in `src/dev-bootstrap/README.md`); until then, production is protected by fail-fast config checks and by the production image stripping this code out (the `CUTOVER-REMOVE` strip runs in all three Dockerfiles: `docker/web`, `docker/worker`, and `docker/image`).
- **Build the admin club-cleanup actions (to retire the override CSV).** Before go-live, club classification fixes (force-keep or force-junk a club) are hand-edited rows in a pipeline CSV, because the database is rebuilt from CSVs on every reload (James owns that CSV procedure, below). The real design is an admin work-queue action that does this against the live database. Building those admin actions retires the temporary CSV approach.
- **Seed the remaining test personas as their features get built.** The test suite seeds a sample user for every account type whose feature exists. The still-missing ones are blocked on features that do not exist yet: event organizers, group owners and members, vote-eligibility lists, and a banned legacy member. Add each persona in the same change that builds its feature.
- **Build search-engine and crawler readiness.** There is no `robots.txt`, no sitemap, and the page layout never outputs a meta description or canonical link (services already shape an `seo.description`, for example `legalService.ts`, but the layout discards it, so this part is half-plumbed rather than greenfield). We want public pages indexed and everything else hidden from search engines. Build: a production `robots.txt` that allows assets and points to the sitemap; a per-page title, meta description, and canonical link in the layout; a way to mark authenticated pages "do not index"; a sitemap; and a "do not index" header on the non-production sites. This is a named go-live blocker.
- **Wire up the accessibility tests.** The testing standard calls for automated accessibility checks (using axe, against the WCAG 2.1 AA standard) on every important page. A few `@a11y`-tagged end-to-end checks exist for keyboard reachability and form-label association on the onboarding flow, but there is no axe-core WCAG scan on any page and `run_all_tests.sh` has no accessibility-only mode. Add axe-core checks across the browser test suite, tag the high-traffic public ones for the quick smoke run, and add an option to `run_all_tests.sh` that runs only them.
- **Source the Stripe webhook secret from AWS, not by hand.** The design says production reads the Stripe webhook signing secret from AWS Parameter Store at startup, like the other secrets. Currently the operator pastes it into the server's env file by hand because no Parameter Store entry exists for it. Add the secret to Parameter Store in both staging and production Terraform, grant the server permission to read it, have the deploy pull it from there, and update the DevOps guide.
- **Actually send the notifications the services promise.** Three services document, in their header contracts, emails and mailing-list updates that the code does not yet send: the membership-tiering service (tier-change and Hall-of-Fame / Big-Add-Posse congratulation emails, plus an admin-alerts mailing-list update when someone becomes admin), the active-player service (a vouch-confirmation email), and the event service (organizer, participant, and sanctioning emails). The email outbox and worker that send these already exist and are tested; the gap is that the service code never enqueues them. Add each send after its database transaction commits. The event-service emails arrive with the event-organizer feature.
- **Build the curator source-of-truth cutover tests.** At go-live the curator content source of truth switches from the `/curated/` JSON sidecars to the persistent production DB (the seeder stops running against production; admins author through the admin UI). Add special tests that pin this switch: curator `media_items`, galleries, and S3 bytes survive a data-preserving `scripts/deploy-migrate.sh` deploy with no seeder run; admin-UI create, edit, and delete is the sole authoring path and is durable without a seeder replay; and the seeder fails fast rather than orphan-deleting admin- and member-created rows if pointed at a persistent production DB. (Migration gate OR9, §29.14; DD §1.13.)

## James — data pipeline and freestyle

James owns the data pipeline (turning the old site's data into the new database) and all freestyle content. This section carries the most detail. The cutover-critical pipeline work comes first, then the freestyle work.

### Data pipeline

- **Finish the legacy member-account import.** This is the cutover-critical pipeline that reads the old site's member dump and loads it into the new member table. The loader already does all the filtering of invalid rows, drops passwords and session tokens, and counts every exclusion; the extract, admin-extract, and validation scripts are all built, tested, and have been dry-run against the real test dump. That dry run turned about 33,665 dump rows into 25,495 importable members, with real names on 100% and country on 99.9%, and matched all 73 admins to a member. What is left before the real production load: (a) write the script that computes each member's starting membership tier, which needs five new columns in the database schema (a change to coordinate with Dave) and needs Steve's committee/board table to know who was on the board; (b) fill in each member's Hall-of-Fame and Big-Add-Posse honor flags by matching them against the local roster files, because right now they are left blank and default to off, which would wrongly deny those members their tier-2 grant; (c) finish cross-checking that the member IDs in the dump match the IDs used in the old profile-page URLs; and (d) load the final dump taken after the site is frozen. The detailed column-by-column mapping is in `docs/MIGRATION_PLAN.md` §15.16.
- **Re-verify the honor rosters before go-live.** The Hall-of-Fame and Big-Add-Posse honor flags are validated against captured roster snapshots (no flag without roster proof), and the snapshots have been reconciled to the live rosters (Hall-of-Fame spelling/casing fixes plus the latest inductees; Big-Add-Posse was clean). Remaining: re-run the read-only live-roster drift check (`diff_live_honor_rosters.py`) once more shortly before go-live in case the public rosters change, and refresh the snapshots if it reports real drift.
- **Optional: tidy a few dangling contact-label residuals in club descriptions.** The contact-detail scrub is settled: a full re-derivation of every description in `legacy_data/seed/clubs.csv` finds no email addresses and no phone numbers (the only digit runs are Facebook group IDs inside URLs, year ranges, and dates; the `@` characters mean "at a location" or sit at an already-stripped value). A prior removal pass left about five descriptions with an empty label where a number or email used to be: `Cell- / Email-` (PAAAFA), `new phone number is .` (AIR-A-ZONA), `email / phone number` (Aurora Footbag Club), and `Text Happy @` (Nevada County Footbag Club). These leak no contact data; cleaning the leftover phrasing is a cosmetic copy-edit that still changes description text, so it needs maintainer approval. Low priority.
- **The club-classification override procedure (the CSV behind Dave's admin-cleanup item).** Before go-live, forcing a club to be kept or junked is done by hand-editing rows in `overrides/club_classification_overrides.csv`. The procedure: look up the club's key by name from the seed file (never guess it), fill in the name and reason so the row explains itself, re-run the club-classification build script and check the override count and the difference from the last run, then run QC. The loader deliberately fails loudly on an unknown category or a club key that no longer exists.
- **Score-text extraction (the rest is deferred).** Done: the kick counts for consecutive-kicks events are now pulled out of the raw result text. Deferred: the free-text "sick" and "routine" descriptors, which are multilingual, sparse, and would mean changing the core text parser for little gain.
- **Workbook parity check (one part deferred).** Done: a check that the event count in the published workbook matches the canonical event list. Deferred: a stricter check that the year sheets and the excluded-events list exactly account for all events, which is not currently computable because the year sheets list event names rather than keys and about 9 sparse events appear on both.
- **Override visibility (database exposure deferred).** Done: a sheet in the workbook listing every event whose results were overridden, plus any stale overrides. Deferred: surfacing an "overridden" flag in the canonical data and the database itself.

### Freestyle

- **Finish freestyle self-containment: relocate the symbolic-grammar inputs into `freestyle/`.** The freestyle-folder lift is done: the pipeline lives in `freestyle/` and rebuilds every table from committed inputs via `freestyle/run_freestyle.sh` (deterministic, no mirror, no network — the footbag.org corpus is a committed snapshot and the live scrape is gated behind `--live`, guarded against any new live pipeline fetch); the legacy pipeline (`reset-local-db.sh` delegates, `run_pipeline.sh` no longer builds freestyle) is decoupled; no runtime/request path reads `legacy_data/`; and the symbolic-grammar layer is loaded into the database (`freestyle/loaders/26_load_symbolic_grammar.py`), so the runtime no longer reads `exploration/`. The `legacy_data/` freeze is therefore no longer blocked by freestyle. One remainder, and it is NOT a freeze blocker (it lives in `exploration/`, not `legacy_data/`): the symbolic-grammar CSVs (`exploration/symbolic-grammar-2/`) and their `build_symbolic_grammar_*` generators still sit outside `freestyle/`; the loader reads the CSVs at build time and the runtime reads the DB, so moving them into `freestyle/` is tidiness, not a correctness gap. Plan + move-map in `exploration/freestyle-folder-lift/`.
- **Source a replacement video for the spinning symposium whirl trick (curator).** The dead-media audit is resolved except one item. Verified by web check: the dead Vimeo world-record video (Ducking Osis, Honza Weber) is gone (HTTP 404) and was a live broken embed, now fixed by clearing the link in `records_master.csv` while keeping the record. The "tombstone" curated video is gone but was already excluded from public view by its `#unavailable_embed` tag, and the trick keeps two other live curated videos, so coverage is intact. The "Lon Smith shred clip" was a stale reference: no such video is flagged dead and the SkyLon Smith videos check out live. The one remainder needs a curator: the "spinning symposium whirl" curated video is gone (private) and is the only video for that trick, so the page now has no demo; it is already hidden by the `#unavailable_embed` tag, but someone should find a live replacement clip to restore coverage. Note: the records loader (`10_load_freestyle_records_to_sqlite.py`) is additive (INSERT OR IGNORE, no DELETE), so edits to existing record rows only take effect on a fresh database build, not an in-place rebuild.
- **Link the remaining 16 world-record videos to their tricks (curator input needed).** Of the 165 record trick names, 16 still do not resolve to any active trick or alias, so those records badge nowhere. The mechanical residual is exhausted: the easy spelling/folk variants are wired, and a clean cascade promotion was minted (`pixie-ducking-eggbeater`, for "Weaving Pigbeater"). The 16 left each need a judgment call, grouped by what is missing: (a) undefined operators - "Motion", "Locomotion" (the "motion" operator), and "Solestice" ("osis flapper") have no registered modifier to build from; (b) unpublished bases with no mirror exemplar - "Scorpion's Tail" and "Spanishfly" name a "double down" base that is not an active trick (only `down-double-down` is); (c) "Double Dyno" and "Double Whip" - "double" is a structural prefix, not a registered operator, and the "(rev)" reading is ambiguous; (d) degenerate structures with no exemplar to mirror - "Gyro Toe", "Toe Spinning Toe", "Blink"; (e) "Stepping Ducking Blurry Whirl" - the folk name (four operators, 6 ADD) and the structured name (two operators, 5 ADD) disagree; (f) ADD conflicts where the record value contradicts the trick its structured name points to - "Blurry Drifter" (record 5 vs `tombstone` 4) and "Enterrage" (record 4 vs `terraging-legover` 5). Separately, three "Unique N-Dex"/"Unique N-ADD" entries are consecutive-dex record categories, not tricks, and correctly badge nowhere.
- **Freestyle trick-promotion decisions (some blocked on an outside expert).** Expanding the trick dictionary further needs several judgment calls from the curator rather than anything mechanical: whether an operator can apply to itself twice, operator-reference entries for "blazing" and "terraging" (defined in the trick-modifier inputs but missing from the authoritative operator reference), a large batch of crossbody tricks (about 47, the single biggest lever), and a couple of unusual cases. There is a 41-row curator adjudication worklist (`legacy_data/freestyle_reconciliation/james_adjudications.csv`) for the rest. Some of this is blocked on a Wave 2 reply from rules expert Red Husted (see Blockers). Separately, a known bug still overcounts the difficulty of three rotational atomic tricks: the `atomic` modifier carries a +2 rotational difficulty bonus where the settled value is +1, so atomic-swirl and atomic-rev-swirl score one too high.

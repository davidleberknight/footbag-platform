# IMPLEMENTATION_PLAN.md

This is the single plan of record for the footbag.org modernization: every open work item and every open question we still need to resolve, grouped by who owns it. It is the clean baseline we will load into the go-live kanban board. Long-term design and rationale live under `docs/`; this file tracks what is left to do.

How to read it. Sections are ordered by owner, from the people we are coordinating with (Steve, then Julie) to the maintainers doing the build (Dave, then James, whose pipeline work carries the most detail). Each item is written to stand on its own: you should not need to open another document to understand what the task is. Every item must be pedantic for a human reader: state the task in full and unambiguously, name exact identifiers (file paths, route strings, schema objects, env vars) rather than vague gestures, give an explicit done-condition, and use no internal shorthand, finding ids, or labels a reader has not been given. An item a human cannot act on without opening another document or guessing what it means is not done. A reference to `docs/MIGRATION_PLAN.md` (the detailed migration design) or a file path appears only where it points at the actual work or saves copying a large table here. Finished items are deleted from this file rather than marked done.

Owners. Steve is the current footbag.org webmaster (he holds the legacy-site, DNS, email, and data-export facts). Julie is the IFPA secretary (she decides IFPA governance, groups, and votes). Dave is the project maintainer and manager (platform code, and he drives every open question to a decision). James is the historical-data and freestyle maintainer (the data pipeline and all freestyle content).

Tags. Each Dave and James item carries a category so this plan can seed the go-live kanban board directly: **[BUG]** a verified defect in running code; **[DEVIATION]** running code intentionally differs from the intended design, guarded, with a removal or unblock condition; **[KANBAN]** a forward feature, test, or gate that becomes a board card; **[PRE-KANBAN]** must be resolved before the kanban board launches and testers engage, to make the baseline trustworthy; **[BLOCKED]** waiting on external input (Steve, Julie, the IFPA, or rules expert Red). Steve's and Julie's items, and the final MVP scope, are open questions, marked **[Q]**. The Blockers, Release gates, and Deferred sections are context, not board cards.

What is and isn't an item here. Documented future work — a capability a canonical doc describes as design intent — is not a gap or a defect; it becomes an item only as a **[KANBAN]** card once scoped for build, and the plan never builds it unsolicited. Only a verified defect (**[BUG]**) or a guarded, intentional difference from the design with an unblock condition (**[DEVIATION]**) is tracked as drift. Current-versus-target status lives here, never in the canonical docs, which stay pure design intent.

### Gaps between what the site promises and what it does

Each item is a place where a shipped user story is not fully delivered. Delete it when the fix ships with a test. The name in parentheses is the user story it belongs to.

- **[KANBAN] Screen new passwords against a breached-password corpus.** Password validation enforces only length (8–128, no complexity), which matches the passphrase-friendly standard, but new passwords are not checked against known-compromised-password lists, which the same standard recommends. Add a breach check at registration and password change that rejects a known-compromised password with a clear message. This is new behavior, so spec the criterion into `V_Register_Account` / `M_Change_Password` first. Use the HaveIBeenPwned Pwned-Passwords k-anonymity range API (privacy-preserving: only a 5-character SHA-1 hash prefix leaves the server, never the password), with a short timeout and fail-open on outage (a breach check is defense-in-depth, not an availability gate); optionally add a small bundled top-N breached-password list as an offline floor so the most common passwords are still blocked during an API outage. Done when new and changed passwords are screened and a firing test covers a known-compromised rejection. (V_Register_Account, M_Change_Password)
- **[DEVIATION] Email content is hard-coded in code generators for now (accepted, to test current email scenarios).** Until the curated-JSON-then-database template pattern above is built, each email's subject and body come from a hard-coded generator in `src/services/emailContent.ts`; the email service's registry maps each `template_key` to its code generator plus its `pii_classification`, and that registry is the initial template set the current email scenarios test against. The generators and the registry carry `Current:` / `Target:` comments. Remove this deviation when the JSON sidecars seed `email_templates` and the email service renders from there.
- **[DEVIATION] The admin email-log page shows message metadata only, not the template body.** The per-message outbound-email log at `/admin/email-log` renders recipient, template, and delivery status, but not the message body. The intended design (the admin system-health story) shows the body as its underlying template with the data-merge fields left clearly unpopulated, so an admin can see what was sent without exposing the recipient's rendered personal data. Unblock: render the template body on the email-log page with its merge fields shown clearly unpopulated.
- **[KANBAN] Curate email templates as committed JSON pre-go-live, then cut over to the database at go-live, exactly like curated media.** Email-template content (`subject_template`, `body_template`, `pii_classification`, enabled flag) lives as committed JSON sidecar files pre-go-live, so templates are version-controlled, curated, and tested before launch; a seeder loads them into the existing `email_templates` table (the same shape as the curated-media `/curated/` sidecars seeded by `seed_fh_curator.py`). At go-live the source of truth switches to the persistent production database: the seeder stops running against production and admins author through the editor below, with the template rows surviving a data-preserving deploy. Capture this design in the canonical docs via doc-sync, a DESIGN_DECISIONS decision mirroring the curated-media source-of-truth decision and a MIGRATION_PLAN cutover gate mirroring the curator gate, and add cutover tests that pin the switch (templates survive a data-preserving deploy with no seeder run; admin-UI authoring is the sole post-go-live path; the seeder fails fast rather than orphan-deleting admin-edited rows). Done when the JSON sidecars seed `email_templates`, the cutover is doc-synced and tested, and the seeder is go-live-aware. (`email_templates` and its `pii_classification` column already exist in the schema with one template seeded; this item is the sidecars, seeder, and cutover, not the schema.)
- **[KANBAN] Give admins a simple email-template editor.** Building on the email service and the curated-JSON-then-database pattern above, let an admin edit an existing template's subject, body, enabled flag, and PII classification through one simple admin screen; the edited `email_templates` row is the live source the email service renders (pre-go-live it is seeded from the JSON sidecars; post-go-live it is the sole source). Each save writes an `audit_entries` row capturing the editing admin and the template changed. Keep it simple: edit existing templates only, no template creation and no arbitrary HTML. (SYS_Send_Email)

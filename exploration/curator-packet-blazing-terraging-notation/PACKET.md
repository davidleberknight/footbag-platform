## Ruling 1 — Is `blazing` the same structural operator as whirling, or distinct?

**Question:** Is blazing merely an alternate name for the same structural operator as
whirling, or does the "opposite-side-terminal" description imply a distinct operational
grammar?

### What the platform currently encodes
- **Holden:** Blazing = `CLIP > OP IN [DEX] > (op side component)`, glossed as "whirling
  with the opposite-side terminal component"; Holden distinguishes blazing from whirling
  **by terminal-component side only**.
- **Platform position:** "the platform does not separate blazing from whirling" — blazing is
  listed as whirling's *related system*, the "op-side-terminal variant."
- **Whirling is a settled `+1`** (pt11) with published op_notation to mirror, e.g.
  `whirling-mirage` (3) = `SET > OP FRONT WHIRL [DEX] > OP IN [DEX] > OP TOE [DEL]` —
  whirling contributes the single leading `OP FRONT WHIRL [DEX]` bracket, the base trick the
  rest, bracket-count == ADD.

### The status of blazing in canon
`blazing` sits in the **Q4 batch listed "not yet ruled"** in resolved canon §E — it has no
formal Red ruling. Every currently-published `blazing-*` row was nonetheless authored under
the platform assumption that **blazing inherits whirling's `+1` contribution**
(`blazing-butterfly` 4, `blazing-drifter` 4, `blazing-illusion` 3, `blazing-legover` 3,
`blazing-mirage` 3, `blazing-paradox-whirl` 5, `blazing-symposium-mirage` 4,
`blazing-torque` 5). So this ruling is in large part a confirmation of a value already
shipped.

### Recommended default: **blazing = whirling op-side-terminal variant, `+1`**
The decisive argument is preservation, not aesthetics:

> Every published blazing row was authored under the assumption that blazing inherits
> whirling's `+1`. **Confirming that assumption preserves all eight published ADD values;
> rejecting it requires re-authoring eight published rows.**

(Structurally this also holds: the only cited distinction — terminal-component side — is a
side choice, not a count-bearing element, so it does not change bracket-count or ADD.)

Under this default the notation backfills as whirling's pattern with the terminal read
op-side:

```
blazing-mirage    (3)  SET > OP FRONT WHIRL [DEX] > OP IN [DEX] > OP TOE [DEL]
blazing-illusion  (3)  SET > OP FRONT WHIRL [DEX] > OP OUT [DEX] > OP TOE [DEL]
blazing-legover   (3)  SET > OP FRONT WHIRL [DEX] > OP OUT [DEX] > OP TOE [DEL]   (op-side terminal: legover's SAME TOE -> OP TOE)
blazing-butterfly (4)  SET > OP FRONT WHIRL [DEX] > SAME/OP OUT [DEX] > OP CLIP [XBD] [DEL]
blazing-drifter   (4)  SET > OP FRONT WHIRL [DEX] > OP IN [DEX] > SAME CLIP [XBD] [DEL]
blazing-torque    (5)  SET > OP FRONT WHIRL [DEX] > OP IN [DEX] > (back) SPIN [BOD] > OP CLIP [XBD] [DEL]
blazing-symposium-mirage (4)  SET > OP FRONT WHIRL [DEX] > (no plant while) OP IN [BOD] [DEX] > OP TOE [DEL]
blazing-paradox-whirl    (5)  SET > OP FRONT WHIRL [DEX] > OP IN [DEX] [PDX] > OP CLIP [XBD] [DEL]
```

Bracket-count == ADD on every row. If you instead rule blazing **distinct** (its own weight
or its own opening, e.g. the set-form `CLIP > OP IN [DEX]` rather than whirling's `SET > OP
FRONT WHIRL [DEX]`), the 8 rows re-author against that and any ADD change propagates.

## Ruling 2 — Which structural element carries terraging's third count?

**Red has already ruled terraging = `+3`** (pt6); the value is not reopened. The unresolved
question is purely notational: which structural element carries the **third count** beyond
the two inward dexes implied by the operator's "double pixie" description.

### Why a third count needs placing
"Double pixie" reads arithmetically as `+2` (pixie `+1`, applied twice; pixie's own notation
is `TOE > SAME IN [DEX] > OP TOE [DEL]`). But Red ruled terraging `+3`, so the notation must
place one count-bearing element beyond the two inward dexes. The Holden set-form gives the
shape but not the resolution: `TOE > SAME IN [DEX] > SAME IN [DEX] >` — two same-side inward
dexes, trailing open for the base trick's terminal.

### The choice

**Option A — third dex (recommended)**
```
SAME IN [DEX]
SAME IN [DEX]
SAME IN [DEX]
```
Contribution = `+3`. Backfill example:
```
terraging-illusion (5)  TOE > SAME IN [DEX] > SAME IN [DEX] > SAME IN [DEX] > OP OUT [DEX] > OP TOE [DEL]
```
— three terraging dexes (`+3`) ahead of illusion's `OP OUT [DEX] > OP TOE [DEL]` (2) = 5,
bracket-count == ADD. Same three-dex lead for `terraging-legover` (5) and `terraging-mirage`
(5).

**Option B — two dexes plus another count-bearing element**
```
SAME IN [DEX]
SAME IN [DEX]
+ unresolved third component   (a distinct element — e.g. a spin/step inherent to the doubling — rather than a third repeated dex)
```
Same `+3` total, but the third count sits as a distinct element you specify.

### What your answer moves
`terraging-illusion` (5), `terraging-legover` (5), `terraging-mirage` (5) — ADD already fixed
at 5; only their `operational_notation` is gated on this. No value moves either way.

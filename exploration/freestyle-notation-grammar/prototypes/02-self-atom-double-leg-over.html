Prototype notes — self-canonical atom

What this prototype is testing

Whether atom-resolved rows present as a single named identity rather than a fake decomposition. The classifier could have wrongly produced something like double + leg + over → unresolved; D1 self-atom recognition collapses that into a clean atom row.

What feels successful

  • The dashed-border atom span visually communicates "this is one indivisible thing" — distinct from modifier-decomposed rows.
  • Empty role rows in the decomposition table aren't padding; they're affirmative ("this row has NO set, NO rotation, NO modifier — it's all atom"). The information density is honest.
  • The "double is NOT a modifier" note inline with the modifier row pre-empts the obvious user question: "but isn't 'double' a count modifier?"
  • The lineage view exposes a real ontology insight — the legover family is dominated by self-atoms, unlike whirl. That's editorially important and surfaces visually.

What feels overloaded

  • The modifier-layering section is essentially trivial for atoms (one box). Could be collapsed to a single line in Phase-3 or omitted entirely for self-atom rows.
  • The "Related tricks" section is weak for atoms — there's no obvious cross-family lens. The "other self-atoms" related list is structurally honest but pedagogically thin (atoms don't share semantic structure beyond being atoms).

Ontology concerns exposed

  • The atom signals "no further structure," but is that always true? double-leg-over IS a productive multiplicity (double + legover) per Red pt3 reading; we just chose not to decompose because the community name persists. The UI risks misleading learners into thinking "double leg over has no decomposition" when in reality "the IFPA dictionary chose not to expose the decomposition." Phase-3 consideration: an "ontology footnote" affordance per atom row.
  • "Tautological exact" status. Self-atom rows always status=exact because computed_adds = asserted_adds by construction. This is technically correct but slightly meaningless as QC. Maybe add a sub-status like exact (self-atom) vs exact (modifier-derived).
  • Display name vs slug. The atom span shows "double leg over" (with spaces) but the internal slug is "double-leg-over" (with hyphens). Currently the prototype shows the display form on the atom span and slug form on the lineage card. Pick one for Phase-3 consistency.

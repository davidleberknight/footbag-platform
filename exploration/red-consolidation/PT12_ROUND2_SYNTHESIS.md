## §3. Decomposition taxonomy — evaluation

### §3.1 The proposed five classes

The maintainer proposed:

1. **Flat compositional decomposition** — Surface name = Modifier₁ + Modifier₂ + … + Base. Each token in the surface name maps to one operator. Example: `Paradox Whirl` = paradox(+1) + whirl(3) = 4.
2. **Expanded semantic decomposition** — Folk single-token decomposes to multi-modifier. Example: `Blurry` = stepping + paradox.
3. **Hidden cross-body / X-dex semantics** — Named operator carries a hidden structural component (X-dex, paradox-from-toe) not visible in its surface name. Example: `Atomic Mirage` = atomic-as-toe-launched-paradox-mirage.
4. **Positional / directional zero-weight operators** — Carry +0 ADD; refine identity rather than difficulty. Example: ss, far, reverse, rooted.
5. **Policy-dependent operators** — Weight depends on base class (rotational vs non-rotational, etc.). Example: atomic = +1/+2 per pt10.

### §3.2 Validation against the new rulings

| New ruling | Best-fit class | Evidence |
|---|---|---|
| Blurry = stepping + paradox | Class 2 (expanded semantic decomposition) | Blurry as folk-name compresses two locked operators |
| Barraging Osis = two dexes + osis | Class 2 OR Class 6 (quantifier-with-explicit-count, see §3.3) | Ambiguous: depends on whether "two dexes" is a *named-stack* or a *count-based shorthand* |
| Atom Smasher carries X-dex from toe | Class 3 (hidden cross-body / X-dex semantics) | Atomic surface name doesn't carry "X-dex" or "paradox"; the structure is hidden |
| Far / reverse = +0 | Class 4 (positional zero-weight) | Direct fit |
| Blistering = Gyro Whirling Set | Class 2 (expanded semantic decomposition) | Blistering as folk-name compresses to gyro + whirling + set-base |
| Mobius ≈ Gyro Torque | Class 2 / federation-equivalence axis (§3.4) | Folk-compression OR cross-system synonym |
| Fairy = legitimate operator | Class 5-adjacent (the operator-vs-trick axis surfaces here) | Fairy is BOTH a 2-ADD base AND a modifier-of-some-weight (Round-2 Q4) |

**Verdict:** the five classes are sound. Every new ruling fits cleanly in one or (rarely) two of them.

### §3.3 Refinement: do we need a sixth class?

**Candidate: "Quantifier-with-explicit-count"** for operators like `barraging` (two dexes), `double` (applied-twice), `triple`. The argument for separating:

- Their semantics carry a *count*, not just a *transformation*. Barraging produces TWO dexes specifically, not "one extra dex" or "a transformed dex."
- Their math is multiplicative-by-count, not additive-by-modifier: barraging-X adds (count-1) dex-cost-units to the base.

**Argument against separating:**

- Class 2 already covers barraging cleanly: `Barraging X = (dex) + (dex) + X` is an expanded semantic decomposition.
- Adding a sixth class adds taxonomic surface area for one structural pattern (count-bearing) that's already captured by Class 2's expansion-shape.

**Recommendation: keep the five-class taxonomy. Note quantifier-with-explicit-count as a *sub-pattern* of Class 2, not a separate class.** The distinguishing feature of count-bearing operators is that their expansion is uniform across bases — `Barraging X` ALWAYS produces two-dex pattern regardless of X. Worth flagging this sub-pattern in §6 (Blurry-class characterization) as a parallel "Barraging-class" entry.

### §3.4 Orthogonal axis: provenance

A pattern not captured by structural classes: **provenance**. Federation-equivalence aliases (Gyro Torque = Mobius; Genesis = Furious Whirl) live in a separate dimension from structural decomposition class. A given name has both:

- a **structural class** (Class 1–5) — what the decomposition shape is
- a **provenance** (IFPA canonical / FM canonical / PassBack folk / community folk / curator-coined) — where the name originated

Mobius is Class 2 (expanded: spinning ss torque → spinning ss miraging osis) AND has IFPA + FM provenance, with Gyro Torque as the FM-side name and Mobius as the IFPA-side name. This isn't a sixth structural class; it's an orthogonal annotation.

**Recommendation:** retain the five structural classes; document provenance as a separate annotation axis. The `FM_MATH_DIVERGENCES.csv` already encodes this for FM names; an IFPA-side provenance field would close the loop.

### §4.4 Doctrine choice for `barraging` parser handling

The maintainer asked: should `barrage` become

- **(a)** explicit operator vocabulary,
- **(b)** shorthand expansion rule, or
- **(c)** decomposition alias?

**Recommendation: (a) explicit operator vocabulary, but with structural-expansion semantics documented inline.** Reasons:

- Barraging is already an entry in `OPERATOR_REFERENCE_ENTRIES` (compound-set category). The new ruling doesn't dissolve it; it clarifies its semantics. Promoting to "shorthand expansion rule" would lose the named-trick identity ("Barraging Osis" stays a recognized compound).
- Shorthand-expansion (b) would force the parser to rewrite `Barraging X` into `(dex) + (dex) + X` at tokenization time, which is a hard architectural change and risks the parser-vs-editorial separation forever-rule.
- Decomposition-alias (c) is too weak: an alias suggests two names for the same thing. `Barraging X` is a named compound, not an alias for `dex dex X`.

**Concrete recommendation for the operator-reference entry post-approval:**

```typescript
{
  slug:           'barraging',
  name:           'Barraging',
  category:       'compound-set',
  oneLineMeaning: 'A set primitive producing two same-direction dexes on the same set. Operational form: CLIP > OP IN [DEX] > SAME IN [DEX] >.',
  decomposition:  '(dex) + (dex)  // two-dex set primitive',
  pendingNote:    null,
  workedExamples: ['Barraging Osis (Baroque) = (dex) + (dex) + Osis = 5'],
  lineageNote:    'Decomposition per the 2026-05-15 ruling.',
  curatorConfirmPending: false,
}
```

This preserves operator identity while documenting the structural semantics. The `decomposition` field gains a count-bearing form `(dex) + (dex)`.

### Q5 — Semantic compression: intentional shorthand or retrospective interpretation?

The compression we've been teaching:

```
Osis → Miraging Osis (= Torque) → Spinning ss Torque (= Mobius)
                                = Spinning ss Miraging Osis (deep form)
```

And now:
```
Mobius ≈ Gyro Torque (2026-05-15)
```

Two readings:

| Option | Reading |
|---|---|
| **A. Intentional shorthand** | The community deliberately compressed long forms into shorter names. `Mobius` is the chosen short form for what the structural reading reveals as `spinning ss miraging osis`. The doctrine: long forms decompress to short forms via explicit operator-substitution rules; teaching this compression is teaching the language as the community uses it. |
| **B. Retrospective interpretation** | Short names like `Mobius`, `Torque`, `Drifter` are original; the long structural readings are post-hoc reconstructions that *illuminate* but do not *author*. The community didn't compress; we're reconstructing. The doctrine: structural readings are pedagogical tools, not the canonical statement of the trick. |
| **C. Mixed** | Some short names are intentional (where the long form predates the short), others are original (where the short name is the only name the community used). Per-token determination. |

This affects pedagogy directly:

- Under **A**, the symbolic-compression flow on `/freestyle/glossary` and the equivalence-chain panels are teaching the language's authoring mechanism. The "Mobius decompresses to spinning miraging osis" framing is canonical.
- Under **B**, the same flow is teaching a *structural lens* over the language; the long forms are not authored names. The "Mobius decompresses to…" framing should be rephrased as "Mobius can be read structurally as…".
- Under **C**, the glossary needs per-token annotation (which compressions are authored, which are retrospective).

The 2026-05-15 Mobius ≈ Gyro Torque ruling is the load-bearing case: did `gyro torque` exist FIRST and the community compress to `mobius`, or did `mobius` exist FIRST and we recognize `gyro torque` as a structural reading?

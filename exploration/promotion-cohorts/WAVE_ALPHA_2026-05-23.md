## Wave Alpha — selection rule

Each Tier 1 promote_with_notation row is scored:

| Signal | Score |
|---|---:|
| Notation in canonical-bracket form (no translation needed) | +3 |
| Notation in fm-parens form (curator translates per dual-convention rule) | +2 |
| Notation only in passback or stanford | +1 |
| Standalone slug (no modifier prefix) | +2 |
| Single-modifier prefix slug | +1 |
| Multi-modifier deep compound | -2 |
| Doctrine divergence = none | +1 |
| Stanford parser-clean | +1 |

Selection: rows with score ≥3, then preferentially fill from
families with ≥3 members (for batch curator review), capped at 30.

## Curator workflow

**For Wave Alpha promotions:**

1. Open `wave_alpha_promotions_2026-05-23.csv` and review by family.
2. For each family, decide whether ALL members deserve canonical rows
   per the canonical-vs-compositional rule (named identity / non-
   decomposable / historical / structural-ambiguity-resolver).
3. For approved members, copy `best_notation` into the active
   `red_additions_2026_*.csv` (or equivalent) as the canonical
   bracket-form. Translate fm-parens → canonical-brackets per the
   dual-convention rule.
4. Stage + commit per the standard workflow.

**For notation-backfill:**

1. Open `wave_alpha_notation_backfill_2026-05-23.csv` and start with
   `doctrine_divergence_tier == none` rows (cleanest).
2. Copy `best_notation` into the `operational_notation` field via
   `red_corrections_2026_04_20.csv` (or equivalent corrections file).
3. Run `python3 scripts/parse_freestyle_notation.py --apply` after
   DB rebuild to refresh structural_parse_json.

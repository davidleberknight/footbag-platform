## The one open rule: where does a trick-level same-side qualifier bind when
## the compound has MORE THAN ONE off-side dex?

The resolver's promoted convention flips the unique off-side dex. The five
held names each carry two OP dexes, so "unique" fails and the name stalls.

**Option A — distributive (global same-side execution).** A bare trick-level
same-side qualifier asserts the SAME configuration for every dex component;
the catch keeps the existing rule (changes only when it is the unique
off-side element, which never fires here since all five catches are already
SAME). The promoted-seven convention is the single-off-side special case of
this rule, so A generalizes the existing precedent rather than replacing it.
A record submitter writing "Eggbeater (ss)" most plausibly means the eggbeater
performed on the same side, not a hybrid chain.

**Option B — structural-base-dex only.** The qualifier targets the structural
base atom's dex and leaves operator-injected dexes at their default. This
extends the fairy/pixie-prefix precedent (qualifier targets the base dex, not
the entry dex), but it is ill-defined for exactly these bases: eggbeater's two
OP OUT dexes are the atomic-injected dex and the legover dex, and which one is
"the base dex" is the ambiguity the resolver hit. It also produces mixed
chains (OP IN then SAME OUT) that no source attests as the meaning of "(ss)".

Both options are collision-free against every active canonical for all five
names (verified):

| Name | Option A config (recommended) | Option B config |
|---|---|---|
| Double Leg Over (ss) | `SET > SAME IN [DEX] > SAME OUT [DEX] > SAME TOE [DEL]` | `SET > OP IN [DEX] > SAME OUT [DEX] > SAME TOE [DEL]` |
| Eggbeater (ss) | `TOE >> SAME OUT [DEX] > SAME OUT [DEX] > SAME TOE [DEL]` | `TOE >> OP OUT [DEX] > SAME OUT [DEX] > SAME TOE [DEL]` |
| Fairy Double Leg Over (ss) | `TOE > SAME OUT [DEX] > SAME IN [DEX] > SAME OUT [DEX] > SAME TOE [DEL]` | `TOE > SAME OUT [DEX] > OP IN [DEX] > SAME OUT [DEX] > SAME TOE [DEL]` |
| Pigbeater (ss) | `TOE > SAME IN [DEX] >> SAME OUT [DEX] > SAME OUT [DEX] > SAME TOE [DEL]` | `TOE > SAME IN [DEX] >> OP OUT [DEX] > SAME OUT [DEX] > SAME TOE [DEL]` |
| Smog (ss) | `TOE > SAME IN [DEX] >> SAME IN [DEX] > SAME OUT [DEX] > SAME TOE [DEL]` | `TOE > SAME IN [DEX] >> OP IN [DEX] > SAME OUT [DEX] > SAME TOE [DEL]` |

**Recommendation: Option A.** It contains the promoted-seven precedent as its
degenerate case, matches the ratified per-component vocabulary distributed
over an unnamed-component trick-level tag, avoids Option B's undecidable
"which dex is the base" regress on operator-injected chains, and yields
configurations a submitter can attest.

## Application once the rule is adopted (mechanical)

1. **The five held record names** become DISTINCT_VARIANT_CANDIDATE with the
   Option-A configs above; being attested record names, they promote as
   `<base> (same side)` canonicals exactly like the 2026-07-02 batch of seven
   (same ADD as the base, derived config, record linkage via alias), and
   their record names resolve.
2. **The far/near-redundant band of the 40.** Where every side-bearing
   component of the compound already sits at the qualifier's asserted side,
   the qualifier is redundant and the ruling supplies the curated equivalence
   the doctrine requires: the EV name collapses to the plain form (alias
   where the plain canonical exists — zulu_legover, zulu_pickup, pogo_mirage
   verified live; audit-tag clear where it does not, with the row moving to
   its authoring gate).
3. **The ss forms among the 40 with a unique off-side dex** resolve by the
   existing promoted convention; those with multiple off-side dexes resolve
   by the adopted rule. No promotion happens in this pass for rows whose
   operator is undefined (flailing, frantic, railing, slicing, surfing,
   shooting: all in the Scoring-paper addendum; their positional tag clears
   and the rows retag to their undefined-operator gate, the same pattern the
   competing-identity session ratified for neutron). Rows on defined
   operators (nuclear, pogo, zulu, fairy, stepping, barraging) retag to
   plain authoring with their configuration recorded.
4. **Pixie DSO (ss)** stays held: its base has no operational notation, so no
   configuration can be derived; the base-notation authoring is its own
   small task and is osis-adjacent (the osis spin-to-catch question is with
   Red in the Notation packet).

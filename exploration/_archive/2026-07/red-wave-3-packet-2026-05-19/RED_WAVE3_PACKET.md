## Q1 -- Operational-notation grammar settling

Three operational-notation conventions need ratification before we publish ~48 currently-held operational notations on canonical rows:

**Q1.A -- Flag order on dex with paradox.** Source corpus (paradoxMoves, blurryMoves, pixieMoves) uniformly writes `SAME IN [PDX] [DEX]` (paradox flag BEFORE dex flag). Modern DB has both orderings live; `blur` was authored as `OP IN [DEX] [PDX]` (PDX after DEX). Which is canonical?

- **A.** Source convention: `[PDX] [DEX]` (modifier flag before action flag).
- **B.** Modern convention: `[DEX] [PDX]` (action flag before modifier flag).
- **C.** Either is valid; flag order doesn't carry semantic weight.

**Q1.B -- Step-marker between dex segments.** Modern operational notation (matador, phoenix, tripwalk, blur) uses doubled-arrow `>>` between dex segments to encode the stepping element. Source corpus uses single `>` throughout. Is `>>` canonical?

- **A.** Yes. `>>` encodes stepping structurally; single `>` is plain sequence.
- **B.** No. Single `>` everywhere; stepping is encoded by adding a [DEX] segment.
- **C.** Both forms valid as alternate notations.

**Q1.C -- Surface-stall operational form.** Modern DB writes surface stalls as `[set] > clipper` (set-prefix bracket + surface name). Source corpus (Clipper entry in fundamentalmoves) writes `CLIP [XBD] [DEL]` (no set prefix; bare surface + flags). Which is canonical for the surface-stall row family?

- **A.** Modern: `[set] > surface`.
- **B.** Source: `SURFACE [XBD] [DEL]` (no set prefix).
- **C.** Both forms valid.

**Q1.D -- The `(plant)` marker.** paradoxMoves + pixieMoves use explicit `(plant)` marker in 6+ source compounds (Paradox Symposium Mirage / PS Whirl / Paradox Ripwalk / Magellan / Dimwalk / Parkwalk / etc.) to indicate when the support leg explicitly plants between segments. Modern ATAM grammar §13 has no `(plant)` marker. Adopt it?

- **A.** Adopt as standard ATAM marker for support-leg-plant disambiguation.
- **B.** Reject; plant-vs-no-plant is implicit in flag/segment patterns.
- **C.** Adopt with restricted scope (only in specific compound contexts).

## Q2 -- Spinning / Gyro / Inspinning axis

The lead cross-cutting question from the audit corpus. 5 source documents touch this; the modern doctrine (`symbolicModifierEducation.ts:150`) says "Spinning = full turn 360°; Gyro = half turn 180°" — but **no source operational notation distinguishes them by magnitude**. Three resolution candidates have emerged:

- **A. Modern is correct (magnitude axis).** Spinning = full turn; Gyro = half turn. The source corpus's operational `(back) SPIN [BOD]` is ambiguous between the two; modern doctrine adds the magnitude distinction that the historical record didn't capture.
- **B. Direction axis only.** Spinning = back-spin; Gyro = back-spin (same axis). The "full vs half" distinction is post-source modern invention; the canonical axis is forward/backward (Job 1995). `Inspinning` = (front) SPIN = forward direction.
- **C. Structural axis (anchor + continuation).** Per footbag-sets source: Spinning = anchor-agnostic back-spin; Gyro = CLIP-anchored back-spin with same-dex continuation; Inspinning = anchor-agnostic front-spin. The three are differentiated by ANCHOR + CONTINUATION rather than magnitude or direction.

**Q2 sub-question -- Inspinning.** Source corpus has 3+ "Inspinning Same Side X" compounds (gyroMoves Q1-Q3) plus 2 "Inspinning Paradox X" compounds (paradoxMoves). Modern doctrine: `inspinning` NOT in `freestyle_trick_modifiers` table. Add it?

- **i.** Add as +1 forward-spin modifier (symmetric with spinning/gyro).
- **ii.** Reject; "Inspinning" is just colloquial naming for front-spin; structurally the same operator as spinning.
- **iii.** Add only if Q2 lands on C (structural axis); skip if A or B.

This question's answer cascades to ~10 currently-pending source compounds + the "Mobius has 'spinning ss torque' as alternate reading" multi-reading framing in the chain registry.

## Q3 -- Pogo / Symposium counting-frame reconciliation

The Add-Categories source (`Add-Categories-move-elements.txt`, 2003 doctrinal source) explicitly states pogo "is usually worth 2 adds more than its non-pogo namesake" — citing pogo as a Clipper-set Symposium Mirage done before another move, adding +1 dex + +1 body (symposium element). Your pt5 ruling treats pogo as +0 modifier.

These reconcile under the source's flag-count-equals-ADD doctrine (Add-Categories source: "Each of these is equal to one Add"):

- Source's "+2 more" = STRUCTURAL contribution counted as operational notation flags (+1 [DEX] from pogo's pre-segment + +1 [BOD] from symposium element).
- Modern pogo=+0 = INCREMENTAL modifier value ABOVE what's already counted via the operational notation.

5 audited pogo-X compounds (Pogo Paradox Mirage / Pogo Paradox Barrage / Pogo Paradox Drifter / Pogo Paradox Eggbeater / Pogo Paradox Whirl) show source ADD systematically +1 above modern multiplier-doctrine derivation — consistent with the counting-frame difference.

**Q3 -- Is this reconciliation correct?**

- **A. Yes — counting-frame reconciliation.** Both pogo=+0 (modifier) AND pogo's structural contribution (+1 dex + +1 body via flags) are simultaneously correct; modifier value is INCREMENT above flag count.
- **B. No — pogo is genuinely +0 with no structural contribution.** Source's "+2 more" framing is wrong; pogo doesn't add dex segments OR body flags structurally.
- **C. No — pogo is genuinely +2 (or some other value).** Modern modifier value should be revised.
- **D. Per-compound.** Pogo-X compounds have varying ADD math; no universal counting-frame doctrine.

If A: workbook governance gate confirms this is the right framing AND the same logic explains the systematic FM ADD divergence on pogo compounds (current 5-row queue).

## Q4 -- Direction-reversal atom doctrine

Your 2026-05-11 ruling established `reverse = +0` positional marker. Subsequent canonicalizations applied this:

- **orbit** (canonical row activated 2026-05-19): orbit = `reverse(+0) + around-the-world(2) = 2 ADD`. CORE-ATOM-CANONICAL-RECONCILE-1 (2026-05-15) treats orbit as one of the 12 core atoms (not as an alias of ATW).
- **mirage / illusion** (both canonical): direction-variant pair; each is its own atom at 2 ADD.
- **legover / pickup** (both canonical): direction-variant pair; each is its own atom at 2 ADD.
- **rev-whirl / rev-up** (both canonical at 3 ADD): folk-named direction variants of whirl; curator 2026-05-19 confirms `reverse whirl == whip` (whip is the DB alias for rev-whirl).

**Q4.A -- General rule for direction-reversed atoms.** Should every direction-reversed core atom (of the 12) be its own canonical row?

- **A.** Yes universal. Each direction-variant atom is canonical (mirage/illusion, legover/pickup, atw/orbit, whirl/whip, etc.). Reverse-X is its own row with its own ADD = base ADD + reverse(0).
- **B.** Yes for the established pairs (the 4 above); future direction-variant atoms need per-row rulings.
- **C.** No — direction-reversal is alias-only beyond the established pairs. Whip should be an alias of rev-whirl (already is); other reverse-X forms should follow the same alias treatment.

**Q4.B -- Pixie 4-primitive dual identity matrix.** Maintainer ruling 2026-05-19: pixie has dual identity as BOTH a +1 set modifier AND a 2-ADD trick form (`TOE > SAME IN [DEX] > OP TOE [DEL]`). This extends to a 4-primitive direction-quadrant matrix:

| Primitive | Direction | Operational form |
|---|---|---|
| Pixie | same-leg + in-dex | TOE > SAME IN [DEX] > OP TOE [DEL] |
| Fairy | same-leg + out-dex | TOE > SAME OUT [DEX] > OP TOE [DEL] |
| Quantum | op-leg + in-dex | TOE > OP IN [DEX] > OP TOE [DEL] |
| Atomic | op-leg + out-dex | TOE > OP OUT [DEX] > OP TOE [DEL] |

All 4 are TOE-anchored 2-ADD tricks with single dex segment, AND each is also a +1 set modifier in the modifier table. Confirm?

- **A.** Confirm dual identity for all 4 primitives.
- **B.** Confirm for pixie + fairy only (named in source 1995-2003 corpus as set primitives); atomic + quantum remain modifier-only.
- **C.** Reject dual identity; pixie etc. are set modifiers, the 2-ADD "trick" form is an artifact.

**Q4.C -- Sailing.** Modern DB has sailing at 2 ADD canonical row (category=set) but NOT in modifier table. footbag-sets source defines Sailing as "(Pixie Illusion): TOE > SAME IN [DEX] > OP OUT [DEX] >" — a set primitive form. Add sailing to modifier table to complete its dual identity?

- **i.** Yes, sailing is a +1 set modifier (matching pixie/fairy/quantum/atomic).
- **ii.** No, sailing is trick-only; doesn't function as a modifier on compounds.
- **iii.** Defer pending Q4.A + Q4.B answers.

## 1. Butterfly family (post-Wave-2: 12 members)

```
ADD 3:  butterfly
ADD 4:  atomic-butterfly, dimwalk, ducking-butterfly, parkwalk, ripwalk,
        sidewalk, spinning-butterfly, [tripwalk ←]
ADD 5:  bigwalk, [matador ←], [phoenix ←]
```

### 1.1 Learning progression entries

- **Tripwalk (4 ADD)** -- quantum's +1 introduction on butterfly. Natural progression from butterfly (3 ADD) base. Sibling 4-ADD compounds for comparison: ducking-butterfly (body compression), spinning-butterfly (body rotation), atomic-butterfly (paradox-atomic), ripwalk (pt11 = stepping butterfly), sidewalk / parkwalk / dimwalk (compound walk variants).
- **Matador (5 ADD)** -- nuclear's +2 weighting; cleanest 2-modifier example. Progression source: any 4-ADD butterfly compound; pt10 ruling provides anchoring.
- **Phoenix (5 ADD)** -- 3-modifier compound. Progression source: pixie-butterfly + ducking-butterfly held cleanly in isolation, OR sidewalk / parkwalk as 1-modifier walk-family compounds at 4 ADD.

### 1.2 Cross-page pedagogical units

Butterfly 4-ADD octet (post-Tripwalk) becomes the densest modifier-comparison tier in the dictionary -- 8 different modifiers attached to butterfly all at the same difficulty value. Recommended editorial framing on each page: "Sibling 4-ADD butterfly tricks: [list of 7 others, each linked]."

Butterfly 5-ADD trio (bigwalk + Matador + Phoenix) is the natural next progression. Editorial framing: "Sibling 5-ADD butterfly tricks: each reaches the same ADD via a different compositional path."

### 1.3 Recommended prereq references (for execution-guide prose)

- Tripwalk -> butterfly (base, 3 ADD).
- Matador -> any 4-ADD butterfly; pt10 nuclear ruling provides math context.
- Phoenix -> pixie-butterfly (4 ADD; if exists) AND ducking-butterfly (4 ADD); 3-modifier compound assembles the two.

### 2.2 Cross-page pedagogical unit (highest editorial yield in Wave-2)

## 5. Curator-authoring guidance for execution-guide prose

When `WAVE2_EXECUTION_GUIDE_DRAFTS.md` learning-notes prose is authored or revised:

1. **Reference siblings by canonical_name** -- linkable; service-side `relatedTricks` already surfaces them. Example: "Pairs with Spender for cross-modifier comparison."
2. **Surface pt-batch rulings sparingly** -- one ruling reference per page is enough. Example: "Pt11 ruled torque = Miraging Osis"; do NOT list multiple pt-batch rulings.
3. **Avoid technical decompositions in prose** -- the editorial-decomposition table already surfaces base + modifier_links. Prose should read for learner audience, not for parser audience.
4. **Acknowledge unfamiliar modifiers** -- nuclear / quantum / tapping deserve a one-sentence orientation in the trick's first appearance for that modifier on that base. Re-orientation on every appearance becomes repetitive.

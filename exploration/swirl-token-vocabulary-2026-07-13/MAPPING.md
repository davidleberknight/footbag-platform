# Swirl-token vocabulary completion — exhaustive occurrence mapping (J6)

**RATIFIED 2026-07-13 (curator).** The proposed rule below is adopted verbatim:
embedded rotational dexes use the separate mandatory register `<side>
FRONT|BACK WHIRL|SWIRL [DEX]`; the IN/OUT register remains the terminal-matrix
vocabulary and must not absorb rotational dex tokens. The ten bare-WHIRL
respellings in the mapping are applied through red_corrections; Blink's BACK
SWIRL stays with the Red rider's path-vocabulary question. The rule's canonical
homes are the Notation paper's settled premises and the whirling entry in the
operator reference; this artifact is the ruling's evidence record.

The decision input for the swirl-token vocabulary ruling: every active-notation
rotational dex token outside the ratified IN/OUT vocabulary, mapped one by one
to a disposition. Counts derive from the live database on the day of writing:
48 occurrences across 48 rows (FRONT WHIRL 31, bare WHIRL 10, FRONT SWIRL 5,
BACK WHIRL 2). Blink's BACK SWIRL is the documented notation exception owned by
the Red rider list and is out of this ruling's scope.

## The question

The ratified clipper-terminal matrix writes the four terminal rotational tricks
in the IN/OUT direction vocabulary: whirl `OP IN`, reverse whirl `OP OUT`,
swirl `SAME OUT`, reverse swirl `SAME IN`, each into its clipper catch. The 48
occurrences below instead write a named rotation token (`FRONT WHIRL`,
`BACK WHIRL`, `FRONT SWIRL`, bare `WHIRL`) as an embedded dex, almost always
the dex injected by a whirling/swirling operator mid-compound. The ruling:
does the matrix vocabulary absorb these tokens, or are they ratified as named
rotation variants?

## The collision proof: absorption destroys identity

Mechanically translating the named tokens to their matrix directions
(FRONT WHIRL to IN, BACK WHIRL to OUT, FRONT SWIRL to OUT, bare WHIRL to IN)
makes these active canonicals byte-identical to other active canonicals:

| Row normalized | Collides with |
|---|---|
| blaze | stepping_mirage |
| bling_blang | sidewalk |
| fender_bender | fairy_ducking_blender |
| symposium_whirling_mirage | miraging_symposium_mirage AND symposium_miraging_mirage |
| whirling_pickup | miraging_pickup |
| whirling_whirl | double_whirl |
| whirling_rake | reverse_whirling_rake (mutual collapse) |

The collisions are structural, not accidental: an embedded whirl-shaped dex and
a plain directional dex travel the same direction axis, so direction alone
cannot distinguish a whirling operator from a miraging, stepping, or doubling
operator. The rotational wrap is real movement content that the IN/OUT axis
cannot carry. Absorption is therefore rejected by evidence, not preference.

## Proposed written rule (one rule, covers every rotational dex token)

1. **The rotational dex register is sanctioned notation vocabulary.** A
   rotational (wrapping) dex is written `<side> <FRONT|BACK> <WHIRL|SWIRL>
   [DEX]`: an explicit side marker, an explicit FRONT/BACK direction qualifier,
   and the shape token. It scores exactly its brackets, like any dex.
2. **The direction qualifier is mandatory.** A bare `WHIRL` or `SWIRL` dex
   token is malformed: it under-specifies direction (the corpus proves the
   failure: reverse_whirling_rake and whirling_rake differ only in the trick
   name today). The ten bare-token occurrences respell within the register per
   the direction their trick name attests; no bracket, ADD, side, or structure
   changes.
3. **The register is for embedded rotational dexes; the matrix vocabulary is
   for the terminal tricks.** Whirl, reverse whirl, swirl, and reverse swirl as
   tricks keep their ratified IN/OUT matrix notations. The register applies
   where a whirling/swirling operator injects a rotational dex mid-compound.
4. **BACK SWIRL currently appears only in Blink**, the documented notation
   exception; whether it stays an exception or becomes an ordinary register
   token is the Red rider's path-vocabulary question and is not decided here.

## Cross-reference for the follow-on family ruling

whirling_reverse_whirl (`SET > SAME IN [DEX] > OP FRONT WHIRL [DEX] > SAME CLIP
[XBD] [DEL]`) carries a reverse-swirl-shaped chassis (SAME IN dex, SAME CLIP
terminal) under a reverse-whirl name, the same cell anomaly the
stepping_reverse_whirl ruling addresses. That is the cell-assignment question,
not a token question; this mapping only fixes its bare token's spelling.

## What ratification changes

Adopting the rule means: no canonical notation is re-derived, no ADD moves, no
row changes identity. The 38 well-formed register tokens are declared sanctioned
vocabulary as written, and the 10 bare tokens gain their mandatory direction
qualifier via red_corrections rows (respellings only). The parser and renderer
already pass these tokens through opaquely, so no code changes ride on the
ruling.
